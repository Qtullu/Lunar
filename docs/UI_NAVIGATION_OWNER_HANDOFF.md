# UI Navigation Owner Handoff

This checklist is the owner-only Content integration and manual verification pass for the Lunar UI Navigation System. The C++ types named below are present in `Source/Lunar`; section 7 records the authoritative owner-verified runtime rows. A successful C++ build alone never changes a row to Verified.

Do not use this handoff to modify `ULunarDraggableWindow`. It remains mouse-only and outside the Lunar navigation hierarchy.

### Prompt Troubleshooting Notes For The Future Tutorial

#### Parent PromptActions changes do not appear in existing buttons

Unreal may retain a serialized local `PromptActions` array on a derived Widget Blueprint or on a placed widget instance. That local override can continue to win after the C++ base class or parent Widget Blueprint changes its default array.

1. Open the affected Widget Blueprint or the screen containing the affected control instance.
2. Select the control whose prompt data is stale.
3. Open `Prompts > Prompt Actions` and review whether the local array contains intentional project-specific entries.
4. Use **Reset to Default** on the complete `PromptActions` property or array row when the control should inherit its parent defaults.
5. Compile and save the affected Blueprint, then reopen or rerun PIE and inspect the resolved prompt.

Resetting replaces the complete local array. Copy or preserve intentional local actions before resetting them.

#### Empty Icon Override or intentionally icon-free prompt

Errors stating that `Icon Override` is enabled but the brush has no Texture or Material resource mean the request contains `bOverrideIcon = true` with an empty brush. The first message may come from live PIE resolution and the second from `ValidateNavigationScope`; they identify the same authoring problem.

- Per-request image: keep `Require Icon` and `Override Icon` enabled, then assign a valid Texture- or Material-backed brush.
- Icon Set image: keep `Require Icon` enabled, disable `Override Icon`, and add exactly one mapping for the resolved key to the active Icon Set.
- Custom presentation needs no image: disable both `Require Icon` and `Override Icon`. The resolved action may still contain display text and semantic metadata, but a missing icon produces no placeholder, log entry, or validation error.
- Custom presentation needs neither image nor display text: use the same two disabled icon flags and leave `Display Text Override` empty. The receiver may still use the action tag, key, device, owner, selection, and enabled metadata or ignore the standard presentation fields entirely.

If an existing button still shows old flag values after a parent change, apply the `PromptActions` **Reset to Default** procedure above before changing individual members. Compile, save, rerun `ValidateNavigationScope`, and confirm a fresh PIE session emits no new diagnostic.

## 3. Example Menu Control Checklist

### Programmatic Restore And Notification Policy

All public state-changing nodes default to `ELunarChangeNotificationPolicy::Notify`. Leave the advanced `Notification Policy` pin at its default for ordinary Blueprint/C++ changes that must behave like a real value change. Choose `Silent` explicitly when loading a saved value into a newly constructed menu and the apply/save callback must not run again.

`Silent` still clamps or normalizes the input, updates getters, refreshes the visible native/Blueprint presentation, and refreshes the accessibility snapshot. It suppresses the owner Blueprint change event, external multicast delegate, feedback, and native accessibility value-change announcement. It is not a deferred update and it is not a visual-only assignment. Real keyboard, gamepad, pointer, and touch interaction always notifies.

For every stateful control tested below, prove both paths before approving its defaults:

1. Call its setter with the advanced pin left at `Notify`; confirm an effective change updates presentation and publishes the expected owner event/delegate exactly once.
2. Call the same setter with `Silent`; confirm presentation and getters update, while the event/delegate, apply/save callback, feedback, and accessibility value-change announcement do not run.
3. Repeat the same effective value with `Notify`; confirm no duplicate publication occurs.
4. If the control replaces a data set or range, force normalization of the current value and repeat the Notify/Silent comparison.

Recommended save-load Blueprint flow: load the saved setting during Pre Construct or Construct, call the control setter with `Silent`, then let later player input use the default `Notify` path to apply and save the new value once. Using default `Notify` during restore is appropriate only when the project intentionally wants to reapply the setting. After upgrading existing Blueprint nodes, use **Refresh Node** and Compile if the advanced policy pin does not appear.

ListView Multi quick reference: mouse uses click to replace, `Ctrl+click` to toggle, `Shift+click` to replace an anchored range, `Ctrl+Shift+click` to add a range, and double click to activate. Keyboard uses the same modifiers with the ListView axis, plus `Ctrl+A` for all, `Ctrl+Space` to toggle Active, and `Enter` to activate. Gamepad uses plain D-pad/left stick to replace, `RB/R1` plus axis for Active-only movement, `LB/L1` plus axis for a range, both shoulders plus axis for an additive range, `X/Square` to toggle, and `A/Cross` to activate. In the Entry Blueprint, style `IsActiveItem` and `IsSelectedItem` through separate cursor/selection visuals; connect `New Is Selected Item` from the Entry visual-state event rather than relying on the owner aggregate delegate.

Per-widget input opt-out quick test: select any Lunar control and expand `Lunar > UI > Navigation > Input`. `Allow Mouse Input`, `Allow Touch Input`, `Allow Keyboard Input`, and `Allow Gamepad Input` are independent and default on. For a mouse-only developer debug-log ListView, leave Mouse on and turn Keyboard plus Gamepad off; set Touch explicitly for the intended platform. Confirm mouse hover/click/wheel/scrollbar still work, keyboard arrows/WASD/Enter and gamepad D-pad/stick/Accept cannot move or activate the ListView's internal item, a direction from that selected ListView transfers to an allowed neighboring widget, forbidden-device prompts and Navigation Selected presentation are absent, and `Dump Active Navigation Graph` reports `M:on`, the chosen Touch state, `K:off`, and `G:off`. `Can Receive Lunar Selection` remains on because it is the master switch used by the allowed pointer channel; turning it off disables Lunar selection for every channel. Scope Back remains a scope command rather than a widget interaction.
